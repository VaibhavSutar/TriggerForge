# TriggerForge: Project Documentation

This document provides a simple, clear overview of the TriggerForge architecture, execution algorithm, and results.

---

## 1. System Blocks Explained

TriggerForge is built using several main blocks. Here is an explanation of what each block does and its inputs/outputs.

### Web UI (Frontend)
*   **Significance & Work:** This is the visual dashboard where users build their automation workflows using a drag-and-drop interface.
*   **Input / Output Generating:**
    *   **Input (i/p):** User clicks, node configurations, and dragged connections.
    *   **Output (o/p):** A JSON document that describes the entire workflow, including a list of nodes and how they connect to each other.

### Server API (Backend)
*   **Significance & Work:** This acts as the middleman. It receives the workflow from the UI, saves it to the database, and tells the Engine when to run it.
*   **Input / Output Generating:**
    *   **Input (i/p):** API requests containing workflow JSON and user credentials.
    *   **Output (o/p):** Database records (saving the workflow state) and signals sent to the Engine to start processing.

### Core Workflow Engine
*   **Significance & Work:** This is the brain of the operation. It reads the workflow JSON step-by-step and decides which actions need to happen and in what order.
*   **Input / Output Generating:**
    *   **Input (i/p):** The workflow JSON map and any initial data (like data from a webhook).
    *   **Output (o/p):** Execution logs and the final computed data after passing through all nodes.

### Connectors (e.g., Email, AI, Webhook)
*   **Significance & Work:** These are the actual "doers." They connect to outside services like sending an email or asking an AI a question.
*   **Input / Output Generating:**
    *   **Input (i/p):** Configurations with resolved variables (e.g., passing a specific text string extracted from the previous step).
    *   **Output (o/p):** The response from the external service (e.g., an AI-generated paragraph).

## 2. Execution Algorithm

When a workflow runs, the Engine uses a step-by-step algorithm to process it. We define this using a mathematical graph model.

### Symbols Introduced
*   Let **$W$** be the Workflow Graph.
*   Let **$N$** represent a single Node in the workflow.
*   Let **$E$** represent an Edge (the connection line between two nodes).
*   Let **$S_{data}$** be the State Data, which holds all the information generated by nodes so far.
*   Let **$Q$** be the Execution Queue (a list of nodes waiting to run).

### Algorithm Steps

1.  **Find the Start:**
    Look at the Workflow graph **$W$**. Find all starting nodes (nodes that have no incoming edges **$E$** or are explicitly marked as 'Triggers').
    *Function:* $StartNodes(W) = \{ N \mid \text{No incoming } E \text{ to } N \}$
    
2.  **Add to Queue:**
    Put all starting nodes into the Execution Queue **$Q$**.
    *Function:* $Q.push(StartNodes)$

3.  **Process Queue Step-by-Step:**
    While the Queue **$Q$** is not empty, do the following:
    *   Take the next node **$N$** out of **$Q$**.
    *   **Read State:** Get any required data from the current State Data **$S_{data}$**.
    *   **Execute Task:** Run the connector logic for node **$N$** using a mathematical execution function: $Output = Execute(N, S_{data})$.
    *   **Save Output:** Add this new $Output$ back into the State Data **$S_{data}$**.
    *   **Find Next Nodes:** Look at all edges **$E$** leaving node **$N$**. Put the connected next nodes into the Queue **$Q$**.

4.  **Finish:**
    When the Queue **$Q$** is empty, the workflow is complete. Return the final State Data **$S_{data}$**.

## 3. Results & Discussion

### Framework & Cases
The framework handles different execution scenarios (cases) natively using the algorithm above:

1.  **Linear Flow (Case 1):** Node A connects to Node B. The queue easily passes data from A directly to B without interruption.
2.  **Conditional Flow (Case 2):** An "If/Else" node checks a mathematical condition (e.g., $Output > 10$). Based on whether the result is `True` or `False`, it puts only the correct subsequent path into the queue.
3.  **Looping Flow (Case 3):** If a node outputs a list of 5 items, the loop mechanism puts the next node into the queue 5 separate times, once for each item.

### Discussion: Why do we get these results?
We get reliable, predictable executions because the system uses a controlled Queue ($Q$) and a central State Data repository ($S_{data}$). Instead of everything trying to run at once (which causes errors), nodes wait their turn in the queue. 
When a node runs, it has guaranteed access to all prior data because the state is centrally managed and resolved right before execution.

### Conclusion
Based on these mechanics, we conclude that the TriggerForge architecture effectively prevents race conditions and data loss. The mathematical queue-based Breadth-First-Search methodology ensures complete, step-by-step execution, making it highly reliable for complex, multi-branching automated workflows.
